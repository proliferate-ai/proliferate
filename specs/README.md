# Proliferate Documentation

Start here before changing the repository. These documents make ownership,
current behavior, and contributor procedures discoverable without relying on
chat history.

## Authority

Documentation artifacts have distinct jobs:

| Artifact | Meaning | Authority |
| --- | --- | --- |
| Current codebase document | Describes behavior and ownership on `main`. This is the default when no other status is declared. | Operating law for its area. |
| Target contract | Describes an accepted destination that is not fully current. It must be labeled `Status: target` and name the current gap. | Governs only an explicitly assigned migration. |
| Draft | An unapproved proposal under `specs/tbd/**`. | Not operating law. |
| Procedure | Describes how to perform and verify a current task under `specs/developing/**`. | Operating law for that task. |
| Generated reference | Reproducible output under `specs/generated/**`. | Evidence generated by its named owner; do not hand-edit. |
| Delivery specification | Founder-approved intent for one PR, stored outside the repository's permanent documentation path. | Governs that PR's intended delta. |

Chats are working context, not authority. Git history is the archive; completed
plans and mutable rollout status do not belong in the canonical read path.

If a delivery specification contradicts a current area document, stop and
report the exact contradiction. Do not silently redefine either one.

The category names and locations above describe the checked-in tree. Use the
indexes rather than guessing a leaf path.

## Read Order

1. Read [`AGENTS.md`](../AGENTS.md) for the source-area router.
2. Read the relevant structure document before editing that source area.
3. Use [`codebase/README.md`](codebase/README.md) to find the shared capability
   or complete product/system contract.
4. Use [`developing/README.md`](developing/README.md) for task procedures.
5. Read the frozen delivery specification for the intended PR delta.

## Writing Rules

- Describe ownership, behavior, failure handling, and proof directly.
- Put source organization in `codebase/structures/**`.
- Put reusable capability contracts in `codebase/platforms/**`.
- Put complete product and engineering domains in `codebase/systems/**`.
- Put task procedures in `developing/**`; do not duplicate system
  architecture there.
- Label target material explicitly and state how it differs from current code.
- Keep implementation progress in the PR or delivery tracker, not in canonical
  documentation.
- Update current documentation in the same PR that changes its behavior.
- Use [`../scripts/check_docs.py`](../scripts/check_docs.py) after documentation
  changes.
