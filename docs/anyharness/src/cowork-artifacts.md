# Cowork Artifacts

Status: authoritative for cowork artifact lifecycle in AnyHarness.

Scope:

- `anyharness/crates/anyharness-contract/**` for cowork artifact wire shapes
- `anyharness/crates/anyharness-lib/src/cowork/**` for artifact lifecycle logic
- `anyharness/crates/anyharness-lib/src/files/**` for artifact write protection
- `anyharness/sdk/**` and `anyharness/sdk-react/**` for cowork artifact client access
- desktop consumers of cowork artifact HTTP endpoints

Use this doc to answer the cowork artifact questions that the general workspace
and session docs do not answer:

- where artifact truth lives
- how artifacts are created and updated
- what the built-in cowork MCP server exposes
- what the desktop reads over HTTP
- what is in scope for v1 and what is intentionally deferred

## 1. Core Model

- Artifacts are **thread-local, file-backed objects** in cowork worktree workspaces.
- There is **no artifact database table** in AnyHarness.
- The durable source of truth is one manifest file per cowork worktree:
  - `.proliferate/artifacts.json`
- Git history is the version history. The manifest tracks only current state.
- Artifact paths are **immutable in v1**.
  - `create_artifact` chooses the path.
  - `update_artifact` may change content, title, and description.
  - There is no rename or move tool in v1.
- Only files listed in the manifest are treated as artifacts.
- Files with renderable extensions but no manifest entry are ordinary files.

## 2. Supported v1 Artifact Types

The first artifact ship supports these file types only:

| Extension | Manifest `type` | Rendering mode |
| --- | --- | --- |
| `.md` | `text/markdown` | inline in runtime |
| `.html` | `text/html` | nested iframe |
| `.svg` | `image/svg+xml` | inline in runtime |
| `.jsx`, `.tsx` | `application/vnd.proliferate.react` | nested iframe after raw JSX transform |

Explicitly out of scope for this phase:

- Mermaid artifacts
- binary artifact blobs
- artifact-initiated MCP tool calls
- artifact version browsing UI
- artifact rename or move
- artifact sharing or publishing

## 3. Ownership In AnyHarness

Keep ownership split the same way the rest of the runtime is split:

- `cowork/service.rs`
  - durable cowork root and thread invariants only
- `cowork/runtime.rs`
  - cross-domain orchestration
  - session startup prompt assembly
  - built-in cowork MCP binding injection
  - turn-end artifact autosave trigger
- `cowork/manifest.rs`
  - manifest schema
  - parse, validate, normalize, persist
- `cowork/artifacts.rs`
  - artifact lifecycle operations
  - create, update, delete, read, list
- `cowork/mcp.rs`
  - built-in cowork MCP server implementation
  - tool registry and dispatch
- `api/http/cowork.rs`
  - cowork artifact HTTP handlers only
- `files/`
  - generic write protection for manifest and artifact-backed paths

Do not put artifact lifecycle logic in HTTP handlers or desktop hooks.

## 4. Manifest File Format

Path:

```text
.proliferate/artifacts.json
```

On-disk JSON shape:

```json
{
  "version": 1,
  "artifacts": {
    "art_a3f1b2c4": {
      "id": "art_a3f1b2c4",
      "path": "dashboards/pr-tracker.html",
      "type": "text/html",
      "title": "PR Tracker",
      "description": "Shows open PRs across my repos",
      "createdAt": "2026-04-10T14:23:00Z",
      "updatedAt": "2026-04-10T14:35:00Z"
    }
  }
}
```

Rules:

- Root `version` is required and must be `1`.
- `artifacts` is an object keyed by artifact id.
- Each artifact entry stores:
  - `id`
  - `path`
  - `type`
  - `title`
  - optional `description`
  - `createdAt`
  - `updatedAt`
- `id` is generated by the lifecycle layer and must be stable for the life of the artifact.
- `path` is worktree-relative.
- `path` must not:
  - be absolute
  - contain `..` path escapes
  - point into `.proliferate/`
- `path` is immutable in v1.
- `type` is derived from the extension, never caller-supplied directly.

## 5. Artifact Lifecycle Invariants

Artifact lifecycle is tool-owned.

- The agent must use artifact lifecycle tools for user-visible artifacts.
- Generic `Write` or `Edit` must not be used for:
  - `.proliferate/artifacts.json`
  - any path currently registered as an artifact in the manifest
- This restriction is required in v1 because artifact `updatedAt` and type/path invariants are manifest-owned.
- Supporting files that are not registered as artifacts may still use normal file tools.

For v1, each artifact is a single file. There is no multi-file artifact bundle contract yet.

## 6. Built-in Cowork MCP Server

Artifacts are managed through a built-in cowork MCP server hosted by AnyHarness.

This is a system-owned server:

- implemented inside AnyHarness
- automatically injected into cowork sessions
- not user-configurable
- not removable

Transport model:

- one session-scoped HTTP MCP endpoint
- one session-scoped bearer token
- the endpoint uses standard MCP `initialize`, `tools/list`, and `tools/call`
- desktop does not call this MCP endpoint directly

Desktop uses HTTP cowork artifact endpoints. The MCP server is the agent control plane.

## 7. Built-in MCP Tool Surface

### `create_artifact`

```text
create_artifact(
  path: string,
  content: string,
  title: string,
  description?: string
)
```

Behavior:

1. validate the path
2. derive `type` from extension
3. ensure the path does not already belong to an artifact
4. write the file
5. load or create `.proliferate/artifacts.json`
6. insert a new manifest entry with generated `art_<uuid>`
7. save the manifest
8. return `{ id, path, type, title }`

### `update_artifact`

```text
update_artifact(
  id: string,
  content?: string,
  title?: string,
  description?: string
)
```

Behavior:

1. resolve the artifact by id
2. if `content` is provided, rewrite the file at the artifact path
3. if `title` or `description` is provided, update manifest metadata
4. bump `updatedAt`
5. save the manifest
6. return `{ id, path, type, title }`

### `delete_artifact`

```text
delete_artifact(id: string)
```

Behavior:

1. resolve the artifact by id
2. delete the file if present
3. remove the manifest entry
4. save the manifest
5. return success

Deletion is idempotent. If the artifact no longer exists, return success.

### `list_artifacts`

```text
list_artifacts()
```

Behavior:

- load the manifest
- validate entries
- return normalized artifact summaries sorted by `updatedAt desc`

### `get_artifact`

```text
get_artifact(id: string)
```

Behavior:

1. resolve the artifact by id
2. validate that the backing file exists
3. read file content as UTF-8 text
4. return normalized metadata plus content

All supported v1 artifact types are text-based, so `get_artifact` returns `content: string`.

## 8. HTTP Read Model For Desktop

The desktop reads artifacts over HTTP, not MCP.

### `GET /v1/workspaces/{workspace_id}/cowork/manifest`

Returns a normalized read model for the artifact panel.

Response shape:

```json
{
  "version": 1,
  "artifacts": [
    {
      "id": "art_a3f1b2c4",
      "path": "dashboards/pr-tracker.html",
      "type": "text/html",
      "title": "PR Tracker",
      "description": "Shows open PRs across my repos",
      "createdAt": "2026-04-10T14:23:00Z",
      "updatedAt": "2026-04-10T14:35:00Z",
      "exists": true,
      "sizeBytes": 14291,
      "modifiedAt": "2026-04-10T14:35:02Z"
    }
  ]
}
```

Rules:

- the HTTP response is a read model, not a raw echo of the file
- artifacts are returned as a sorted array, not the on-disk id-keyed object
- the response may enrich entries with:
  - `exists`
  - `sizeBytes`
  - `modifiedAt`
- artifacts are sorted `updatedAt desc`

### `GET /v1/workspaces/{workspace_id}/cowork/artifacts/{artifact_id}`

Returns one artifact plus content.

Response shape:

```json
{
  "artifact": {
    "id": "art_a3f1b2c4",
    "path": "dashboards/pr-tracker.html",
    "type": "text/html",
    "title": "PR Tracker",
    "description": "Shows open PRs across my repos",
    "createdAt": "2026-04-10T14:23:00Z",
    "updatedAt": "2026-04-10T14:35:00Z",
    "exists": true,
    "sizeBytes": 14291,
    "modifiedAt": "2026-04-10T14:35:02Z"
  },
  "content": "<!doctype html>..."
}
```

Error behavior:

- `404` when the artifact id is unknown
- `409` when the manifest entry exists but the file is missing or invalid for its declared type

## 9. Session Startup Policy

Cowork sessions must automatically receive:

- the built-in cowork MCP server binding
- the cowork artifact startup prompt append

The artifact-specific prompt must tell the agent:

- use `create_artifact`, `update_artifact`, and `delete_artifact` for user-visible artifacts
- never edit `.proliferate/artifacts.json` directly
- do not use generic file writes on artifact-backed paths
- use normal file tools for supporting files only

For JSX artifacts, the prompt must also say:

- export a default component
- no required props
- use only allowlisted imports

The prompt is session launch policy and must be applied on both:

- ACP `new_session`
- ACP `load_session`

## 10. Turn-End Autosave

Cowork artifact changes are saved at turn end, not on each tool call.

When a cowork turn completes:

1. detect worktree dirtiness
2. if dirty, stage all changes
3. create one auto-commit covering:
   - artifact file changes
   - manifest changes
   - any supporting file changes made during the same turn

Cloud push remains out of scope for this phase.

## 11. Desktop Expectations

Desktop behavior depends on these contracts:

- fetch manifest on:
  - thread open
  - turn end
  - manual Refresh
- fetch one artifact by id when the user opens it
- use artifact `type` to choose the viewer renderer
- treat cowork artifact UI as thread-local only

Desktop must not:

- parse `.proliferate/artifacts.json` directly
- synthesize artifact ids or types client-side
- mutate artifact files or the manifest directly

## 12. Explicit Non-Goals For This Phase

- Mermaid artifacts
- artifact rename or move
- artifact bundles or multi-file build outputs
- artifact-initiated MCP calls
- publish/share links
- artifact version browser
- desktop polling or file watching for artifact refresh
