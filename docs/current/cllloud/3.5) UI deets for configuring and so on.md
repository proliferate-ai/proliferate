## High level UI model

UI should expose a few reusable configuration primitives everywhere:

- MCP/skill/plugin library
- Agent auth library
- Sandbox auth selector
- Agent run config selector
- Repo/environment config
- Runtime readiness/status

Internally, most config is sandbox-scoped. Externally, avoid making users think
about "sandbox profiles" unless needed. Personal users see "Cloud". Admins see
"Team work" or "Shared cloud".

## Core pages

### MCPs, Skills, Plugins

Purpose: configure tool and skill capability for personal and shared sandboxes.

Rows show:

- name, kind, owner, status;
- enabled for personal;
- public to org;
- required credentials;
- last runtime apply status.

Actions:

- connect MCP;
- enable/disable skill/plugin;
- make public/private;
- reconnect auth;
- view where used.

### Agent Authentication

Purpose: create credentials and choose auth per sandbox/harness.

Sections:

- Proliferate managed credits;
- organization credentials;
- personal credentials;
- shared personal credentials.

Harness-first selector:

```text
Claude    -> choose synced auth or gateway credential
Codex     -> choose synced auth or gateway credential
OpenCode  -> choose supported gateway/synced auth
Gemini    -> synced auth until gateway path is proven
```

Do not show irrelevant provider forms for a harness. Example: Claude setup
should show Anthropic/Bedrock/synced Claude options, not generic OpenAI API key
as a primary choice.

Hosted cloud V1 exact behavior:

```text
show:
  Proliferate managed credits status/action
  synced native auth setup/sync status
  owner share/consent controls for personal synced auth used by shared sandboxes
  per-harness sandbox selector

hide:
  Bedrock AssumeRole gateway form
  Anthropic API key gateway form
  OpenAI API key gateway form
  OpenAI-compatible gateway form
  model-deployment/provider-routing controls
```

The UI can still display a BYOK credential or selection that already exists in
the database, but only as unavailable/disabled. This matters for private test
deployments and stale rows:

```text
selected BYOK credential exists but hosted BYOK disabled
  -> selector value shows "unavailable in hosted cloud"
  -> user cannot save it again
  -> user can choose managed credits or synced auth instead
```

Managed credits should be presented as a credit/status capability, not as an
ordinary credential that admins revoke from the generic credential row. If the
org should stop using managed credits, that should happen through entitlement,
budget, or sandbox selection changes.

Shared synced auth needs explicit consent in the UI:

```text
owner shares personal synced Claude auth with org
  -> row shows source owner
  -> admin can select it for shared sandbox
  -> revocation invalidates shared selection and queues cleanup/refresh

admin sees unshared personal synced auth
  -> cannot select
  -> CTA points to owner share/consent, not admin override
```

### Repo / Environment Config

Purpose: configure how a repo runs on local, personal cloud, and shared cloud.

Fields:

- default branch;
- base branch behavior;
- setup/install command;
- run command;
- env files to sync;
- tracked config files;
- cloud-specific overrides.

Personal env files sync only to personal sandboxes. Shared env files must be
admin-created/shared explicitly.

### Personal Cloud

Purpose: user's cloud sandbox readiness.

Shows:

- target online/readiness;
- MCP/skill/plugin status;
- agent auth status;
- repos enabled for cloud;
- current usage/credits if relevant.

### Remote Access / Dispatch

Purpose: make an existing workspace/session visible and optionally controllable
from Cloud surfaces such as web and mobile.

This is not "copy to cloud." It creates or updates exposure/projection for the
existing runtime.

Primary actions:

```text
Continue remotely
  creates exposure
  upgrades session projection to live
  sets commandable according to owner policy
  opens web/mobile Cloud session

Disable remote access
  pauses/revokes exposure or commandability
  leaves local AnyHarness workspace/session untouched

Move to another target
  separate migration flow
  transfers runnable state
```

UI states:

```text
Not exposed
  visible only on the direct Desktop/target surface

Exposing
  Cloud created exposure; worker is backfilling projection

Live from Cloud
  projection active; Cloud clients can render live state

Read-only from Cloud
  projection active; commandable = false

Unavailable
  worker offline, projection stale, or policy revoked
```

If Desktop and Cloud are both open to the same exposed session, both should show
the same conversation. Desktop may update first because it is connected directly
to AnyHarness; Cloud catches up through worker projection.

### Shared Cloud

Admin-only purpose: org shared sandbox readiness.

Shows:

- shared MCPs/skills/plugins available;
- selected agent auth per harness;
- shared repo/environment config;
- Slack/automation readiness;
- target/worker status.

### Agent Run Configs

Purpose: save reusable "how should this agent run?" choices for chat,
automations, Slack, web, mobile, and Desktop.

The catalog owns available options. The saved row owns selected values.

```text
agent_run_config
  id
  owner_scope: system | personal | organization
  owner_user_id
  organization_id
  name
  agent_kind: claude | codex | opencode | gemini | cursor
  model_id
  control_values_json
    # examples:
    # mode: default
    # effort: high
    # fast_mode: off
    # collaboration_mode: agent
  usable_in_personal_sandboxes
  usable_in_shared_sandboxes
  status
  created_at
  updated_at
```

UI behavior:

- load `catalog.json`;
- choose agent first;
- render only controls supported by that agent and surface;
- store only chosen values;
- resolver later turns chosen values into `CreateSession` fields and live config
  updates.

Do not create separate provider/model config objects for Slack, automations,
web, or mobile. They should all select an `agent_run_config`.

### Automations

Create/edit:

- owner: personal or team;
- target: local, SSH, personal managed cloud, shared cloud;
- schedule;
- repo/workspace;
- agent run config;
- prompt.

Team automation should show the shared sandbox capability summary and link to
MCP/auth/shared cloud settings.

### Slack Bot

Admin-only setup:

- install/connect Slack workspace;
- repo routing: fixed repo or auto-select from allowed repos;
- default agent run config;
- shared sandbox readiness;
- allowed channels/users if needed.

Slack threads become Cloud sessions/workspaces that can later be claimed.

## Reusable components

```text
CredentialPicker
  filters by harness, owner, sandbox type, readiness

AgentRunConfigSelector
  agent kind, model, catalog-backed controls

RuntimeReadinessPanel
  target online, runtime config revision, agent auth revision, worker version

PublicCapabilityList
  public MCPs/skills/plugins available to team work

AgentRunConfigSelector
  loads catalog controls, filters by surface, saves selected values only

WhereUsedDrawer
  sandboxes, automations, Slack configs using this item
```

## UX rules

- Always show source owner for shared credentials.
- Shared sandbox selection is admin-only.
- Personal synced credentials require owner opt-in before shared use.
- Status should fail closed: ready, pending, needs reconnect, needs resync,
  invalid, budget exhausted.
- Keep "make public" copy explicit: public items can be used by team
  automations, Slack, and shared cloud work.
- Changing shared config should show which live/next launches are affected.

## End to end UI flows

Enable personal cloud:

1. User connects Git/repo.
2. User chooses/syncs agent auth.
3. User enables MCPs/skills/plugins.
4. Cloud target initializes.
5. Readiness panel shows config applied.

Configure shared cloud:

1. Admin enables shared cloud.
2. Admin chooses shared agent auth per harness: managed credits, organization
   synced/gateway credential if enabled, or explicitly shared personal synced
   credential.
3. Admin makes MCPs/skills/plugins public.
4. Admin configures shared repo env.
5. Shared readiness becomes ready for automations/Slack.

Create team automation:

1. User/admin chooses Team.
2. UI uses shared sandbox config.
3. UI shows public MCPs/skills and selected agent auth.
4. Schedule saves only automation-specific fields.

Configure Slack:

1. Admin installs Slack app.
2. Admin chooses repo routing behavior.
3. Admin chooses default agent run config.
4. Slack messages create shared cloud sessions.
