# Session Portability: Claude Code

## Bottom Line

A Claude Code session is a single append-only JSONL file
(`~/.claude/projects/<sanitized-cwd>/<session-uuid>.jsonl`) plus a handful of
sidecar directories. The file is self-contained for conversation replay:
every message, metadata entry, compaction boundary, file-history snapshot,
attribution snapshot, and content-replacement record lives as a line in that
JSONL. No external database or index exists.

Session lookup is **cwd-sensitive** -- the sanitized absolute path of the
working directory becomes the parent directory name. Tool calls inside the
transcript embed **absolute paths** (e.g. `file_path` in Read/Edit/Write
tool_use blocks). These two facts are the main obstacles to cross-machine
portability.

Moving a session to another machine requires:
1. Copying the `.jsonl` file (and optional sidecar dirs) to the target's
   `~/.claude/projects/` tree.
2. Ensuring the target's working directory hashes to the same directory name,
   OR passing the `.jsonl` path directly to `--resume`.
3. Accepting that absolute paths inside tool events will reference the
   original machine's filesystem.

Collection should only happen for a quiescent session. Exporting an actively
appending JSONL risks capturing a partial turn or unresolved tool-use/tool-result
pair.

## Resume Path

Two CLI flags trigger resume:

| Flag | Behavior |
|------|----------|
| `-c` / `--continue` | Loads the most recent session from `~/.claude/projects/<sanitized-cwd>/` |
| `-r <uuid-or-search>` / `--resume <uuid-or-search>` | Loads a specific session by UUID, by search term (custom title match), or by direct `.jsonl` file path |

Entry flow (all references relative to `~/claude-code/`):

1. `src/main.tsx` action handler, line ~991: defines `-c, --continue` and
   `-r, --resume [value]` via Commander.
2. `src/main.tsx` line ~3101: `if (options.continue)` calls
   `loadConversationForResume(undefined, undefined)`.
3. `src/main.tsx` line ~3355: `else if (options.resume)` resolves the
   argument to a session UUID or `.jsonl` path, then calls
   `loadConversationForResume(sessionId | log, sourceJsonlFile)`.
4. `src/utils/conversationRecovery.ts` `loadConversationForResume()`:
   - `undefined` source: calls `loadMessageLogs()` -> picks most recent.
   - `string` source: calls `getLastSessionLog(uuid)`.
   - `sourceJsonlFile` path: calls `loadMessagesFromJsonlPath(path)`.
   - Deserializes, filters unresolved tool_uses, detects mid-turn
     interruptions, runs `SessionStart` hooks with source `'resume'`.
5. `src/utils/sessionRestore.ts` `processResumedConversation()`:
   - Calls `switchSession(sessionId, projectDir)` to update global state.
   - Calls `restoreSessionMetadata()`, `adoptResumedSessionFile()`.
   - Restores agent definition, coordinator mode, worktree state, context
     collapse, cost state.
6. The interactive `/resume` slash command
   (`src/commands/resume/resume.tsx`) offers a picker UI and calls the same
   resume pipeline.

`--fork-session` creates a new session ID while keeping the resumed
transcript as initial messages.

## Durable Identity

**Session ID**: Standard UUID v4, generated by `randomUUID()` in
`src/bootstrap/state.ts` line ~447. Stored as `STATE.sessionId` (type
`SessionId`, which is a branded `string`).

**Format**: `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx` (lowercase hex,
validated by `uuidRegex` in `src/utils/sessionStoragePortable.ts` line 23).

**Generation**: happens once at process startup in `getInitialState()`.
`regenerateSessionId()` creates a new one on `/clear`. `switchSession()`
replaces it on `--resume`/`--continue` (non-fork path).

**In-transcript**: every JSONL line of type `user`, `assistant`,
`attachment`, or `system` carries `sessionId` as a string field
(`SerializedMessage` type in `src/types/logs.ts` line 8). Metadata lines
(`custom-title`, `tag`, `agent-setting`, `mode`, `worktree-state`,
`pr-link`) also carry `sessionId`.

## Required Artifacts

### Primary: the JSONL transcript

```
~/.claude/projects/<sanitized-cwd>/<session-uuid>.jsonl
```

This single file contains the entire conversation. Line types:

| Line type | Purpose |
|-----------|---------|
| `user` | User messages (text, tool_result blocks) |
| `assistant` | Model responses (text, thinking, tool_use blocks) |
| `attachment` | File/directory observations, hook context, invoked skills |
| `system` | Warnings, info, compact_boundary markers |
| `summary` | Auto-generated conversation summaries (keyed by `leafUuid`) |
| `custom-title` | User-set session name |
| `ai-title` | AI-generated session title |
| `tag` | Searchable tag |
| `agent-name` / `agent-color` | Display name and color for the session |
| `agent-setting` | Which agent definition was used |
| `mode` | `coordinator` or `normal` |
| `worktree-state` | Worktree session data for resume |
| `pr-link` | Linked GitHub PR |
| `file-history-snapshot` | Per-message file state for undo |
| `attribution-snapshot` | Per-file attribution state (ant-only) |
| `content-replacement` | Records of large tool results replaced with stubs |
| `marble-origami-commit` | Context-collapse commit entries |
| `marble-origami-snapshot` | Context-collapse staged state |
| `last-prompt` | Last user prompt (for `claude ps` display) |
| `task-summary` | Periodic agent activity summary |

Compaction: when the transcript grows large, a `compact_boundary` system
message is appended. On resume, `readTranscriptForLoad()` in
`src/utils/sessionStoragePortable.ts` discards everything before the last
non-preserved boundary, reading only post-boundary content. Pre-boundary
metadata is recovered by a separate byte scan (`scanPreBoundaryMetadata`).

### Sidecar: subagent transcripts

```
~/.claude/projects/<sanitized-cwd>/<session-uuid>/subagents/agent-<agent-id>.jsonl
~/.claude/projects/<sanitized-cwd>/<session-uuid>/subagents/<subdir>/agent-<agent-id>.jsonl
```

Each subagent (AgentTool invocation) gets its own JSONL in a subdirectory
named after the parent session UUID. Metadata sidecar:
`agent-<agent-id>.meta.json` (contains `agentType`, optional
`worktreePath`, optional `description`).

### Sidecar: remote agent metadata

```
~/.claude/projects/<sanitized-cwd>/<session-uuid>/remote-agents/remote-agent-<task-id>.meta.json
```

Identity for remote (CCR) agent tasks. Contains `taskId`, `sessionId`,
`title`, `command`, `spawnedAt`.

### Sidecar: tool result persistence

```
~/.claude/projects/<sanitized-cwd>/<session-uuid>/tool-results/
```

Large tool results that exceed the inline threshold are written to files
here. The in-transcript `tool_result` block is replaced with a stub
referencing the persisted file. The `content-replacement` JSONL entries
record which `tool_use_id`s were replaced so resume can reconstruct the
correct in-context representation.

### Sidecar: plans

```
~/.claude/projects/<sanitized-cwd>/plans/<slug>.md
```

Plan-mode artifacts. `copyPlanForResume()` copies the plan to the current
project dir when resuming cross-project. The plan slug is stored per-session
in the `planSlugCache` map (in-memory only, keyed by session ID, with the
slug also persisted in `SerializedMessage.slug`).

### Global config (cost state)

```
~/.claude/.claude.json  (or ~/.claude/.config.json legacy)
```

`projects[<path>].lastSessionId` / `.lastCost` / `.lastModelUsage` etc.
`restoreCostStateForSession()` in `src/cost-tracker.ts` reads this on
resume to restore cumulative cost counters. Not required for conversation
replay, only for `/cost` display accuracy.

### Concurrent session PID files (ephemeral)

```
~/.claude/sessions/<pid>.json
```

Written by `registerSession()` in `src/utils/concurrentSessions.ts`.
Contains `pid`, `sessionId`, `cwd`, `kind`, agent name. Cleaned up on exit.
Not needed for portability -- purely for `claude ps` display.

## Lookup Sensitivity

### Directory path hashing

The project directory for a session is computed by:

```
sanitizePath(canonicalizePath(cwd))
```

Where `sanitizePath` (`src/utils/sessionStoragePortable.ts` line 311)
replaces all non-alphanumeric characters with hyphens:

```
/Users/pablo/myproject  ->  -Users-pablo-myproject
```

For paths longer than 200 characters, a hash suffix is appended. The hash
function differs between Bun (`Bun.hash`) and Node.js (`djb2Hash`), which
means the same long path produces different directory names depending on
the runtime. `findProjectDir()` handles this by falling back to
prefix-based directory scanning. In practice, direct-path resume
(`claude --resume /path/to/file.jsonl`) largely sidesteps this portability
trap because it bypasses project-directory lookup entirely.

`canonicalizePath()` resolves symlinks via `realpath` and NFC-normalizes.

### cwd is baked into the storage path

When you run `claude` from `/Users/pablo/myproject`, sessions go to:
```
~/.claude/projects/-Users-pablo-myproject/
```

Running `claude --continue` from a different directory will look in that
directory's project folder, not the original one. Cross-directory resume
requires:
- Using `--resume <uuid>` with the `/resume` all-projects toggle, or
- Passing the `.jsonl` file path directly to `--resume`, or
- `cd`-ing to the original directory first.

### Worktree fallback

`resolveSessionFilePath()` in `src/utils/sessionStoragePortable.ts` line 403
has a worktree fallback: if the session is not found in the current
directory's project folder, it scans sibling git worktrees.

## Transcript / Tool History Path Sensitivity

### Every message carries `cwd`

Each `SerializedMessage` includes a `cwd` field set to `getCwd()` at write
time (`src/utils/sessionStorage.ts` line ~1059). This is the absolute
working directory path.

### Tool call inputs contain absolute paths

Tool calls persisted in assistant `tool_use` blocks embed absolute paths:

- `FileReadTool`: `input.file_path` = absolute path
- `FileEditTool`: `input.file_path` = absolute path
- `FileWriteTool`: `input.file_path` = absolute path
- `BashTool`: `input.command` often contains absolute paths
- `GlobTool`, `GrepTool`: `input.path` = absolute path

Tool results in user `tool_result` blocks also contain absolute paths in
their text content (file contents prefixed with path, error messages, etc.).

### Attachment messages contain absolute paths

`attachment.filename`, `attachment.path`, `attachment.skillDir` are absolute.
`migrateLegacyAttachmentTypes()` in `src/utils/conversationRecovery.ts`
computes `displayPath` as `relative(getCwd(), absolutePath)` at resume time.

### Impact on portability

If a session is moved to a machine where `/Users/pablo/myproject` is at
`/home/pablo/myproject`, the model will see tool events referencing the old
absolute paths. The conversation history is still loadable and the model can
reason about it, but:
- File-history snapshots reference old paths (undo may fail).
- The model may attempt to use old absolute paths in new tool calls.
- Attachment `displayPath` computation will produce wrong relative paths
  unless cwd matches.

## Minimal Export Spec

To export a session for replay on another machine:

0. Ensure the session is quiescent. Do not read the JSONL while Claude Code is
   still appending to it.
1. **Required**: `<session-uuid>.jsonl` -- the primary transcript.
2. **If subagents were used**: `<session-uuid>/subagents/` directory tree
   (`.jsonl` and `.meta.json` files).
3. **If remote agents were used**: `<session-uuid>/remote-agents/` directory
   (`.meta.json` files).
4. **If tool results were persisted**: `<session-uuid>/tool-results/`
   directory. Without these, resumed sessions will see
   `[Old tool result content cleared]` stubs for large outputs, but the
   `content-replacement` entries in the JSONL let the resume path classify
   them correctly for prompt cache stability.
5. **If plan mode was used**: `plans/<slug>.md` file. Without it, the session
   still loads but plan context is lost.

NOT required:
- PID files (`sessions/`)
- Global config (`.claude.json`) -- only affects cost display
- Any state in `~/.claude/settings.json` or `.claude/settings.json` --
  these are user/project preferences, not session state

## Minimal Install Spec

To install an exported session on a target machine:

1. Determine the target working directory (e.g. `/home/user/myproject`).
2. Compute `sanitizePath(canonicalizePath(targetCwd))` to get the directory
   name (replace all non-alphanumeric chars with hyphens).
3. Place files under `~/.claude/projects/<sanitized-target-cwd>/`:
   ```
   <session-uuid>.jsonl
   <session-uuid>/subagents/...   (if applicable)
   <session-uuid>/remote-agents/... (if applicable)
   <session-uuid>/tool-results/...  (if applicable)
   plans/...                        (if applicable)
   ```
4. Resume with `claude --resume <session-uuid>` from the target directory.

Alternatively, skip the directory placement entirely:
```
claude --resume /path/to/exported/<session-uuid>.jsonl
```
This bypass works because `--resume` accepts a direct file path (the
`.jsonl` suffix triggers the `sourceJsonlFile` code path in
`loadConversationForResume`).

## v1 Recommendation

For AnyHarness session mobility (moving a session from one runtime to
another), the pragmatic v1 approach:

1. **Copy the JSONL + sidecars**: the session is fully self-contained in
   the file tree rooted at `<session-uuid>.jsonl` plus its sibling
   `<session-uuid>/` directory. Tar this up.

2. **Use direct file path resume**: place the tarball content anywhere on
   the target and resume with `claude --resume /path/to/<session-uuid>.jsonl`.
   This avoids needing to match the sanitized cwd directory name.

3. **Accept path mismatch**: the model will see old absolute paths in
   history. In practice this is mostly harmless -- the model adapts to the
   current cwd for new tool calls. Broad transcript rewriting is intentionally
   out of scope for v1.

4. **Skip cost state**: cost counters are cosmetic. If needed, the global
   config's `projects[path].lastSessionId` / `lastCost` can be manually
   set, but this is low priority.

5. **Session ID stability**: keep the original UUID. The `--resume` path
   calls `switchSession(originalId)` and the resumed session continues
   appending to the same file. If you need a fresh ID, use `--fork-session`
   with `--resume`.

6. **Compaction awareness**: large sessions may have compact_boundary
   markers. The resume path handles these transparently --
   `readTranscriptForLoad` skips pre-boundary content. No special handling
   needed for export.

What v1 does NOT solve:
- Live session handoff (the JSONL is append-only; concurrent writes from
  two processes would corrupt it).
- Automatic path rewriting for cross-machine filesystem layout differences.
- Restoration of ephemeral process state (MCP server connections, LSP
  state, terminal sessions).
- Cross-version compatibility if source and destination Claude Code builds
  diverge materially in transcript format or resume behavior.
